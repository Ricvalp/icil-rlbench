Important existing behavior
---------------------------
The JAX runner currently relies on the RLBench cached dataset infrastructure. It builds tasks using:

    QueryMemoryTaskBuilder
    PreparedQueryMemoryTaskBatchIterable

from:

    icil.models.maml.query_memory_tasks

The prepared JAX host batch expected by `icil_jax_query_memory/data/adapter.py` currently contains:

    inner:
        query_xyz
        query_state
        query_valid
        target_action
        optional query_rgb
        optional query_mask_id

    query:
        query_xyz
        query_state
        query_valid
        target_action
        optional query_rgb
        optional query_mask_id

    meta:
        support_ids
        query_episode_id

For Meta-World we will not use point clouds. The first goal is to test GradMem-style and MAML/FOMAML-style memory adaptation in a low-dimensional state setting. This should remove RLBench perception and point-cloud tokenization from the problem.

High-level goal
---------------
Implement a Meta-World data generation and loading pipeline for imitation learning and in-context/meta-learning experiments.

For now, implement only:

1. Expert demonstration generation for Meta-World.
2. HDF5 cache writing.
3. HDF5 cache reading.
4. Dataset/task sampling compatible with the existing query-memory training style.
5. Basic inspection/debug scripts.

Do not implement a new training method yet. The training setup will reuse the existing JAX query-memory GradMem/MAML/FOMAML path first, then later a PyTorch encoder-based path.

Scientific purpose
------------------
This Meta-World pipeline is a diagnostic benchmark for the following question:

    Can gradient descent write useful task information into memory tokens from support trajectories?

This is different from the RLBench point-cloud work. In RLBench, the model has to solve point-cloud perception, object localization, task inference, and action prediction. In Meta-World, we want a low-dimensional control setting where we can test GradMem-style memory writing with fewer confounders.

Meta-World observations
-----------------------
Use low-dimensional Meta-World observations, not images and not point clouds.

Store the raw environment observation, but expose filtered variants for the model.

Required stored arrays per episode:

    obs_raw:        [T, obs_raw_dim]
    obs_model:      [T, obs_model_dim]
    actions:        [T, action_dim]
    rewards:        [T]
    success:        [T] or [T, 1]
    done:           [T]
    trunc:          [T]
    task_name:      string metadata
    task_index:     int metadata
    env_id:         string metadata
    seed:           int metadata
    episode_id:     int metadata

Recommended observation variants to store or derive:

    obs_raw
        The exact observation returned by Meta-World.

    obs_no_task
        Observation with any one-hot task id removed.

    obs_no_task_no_goal
        Observation with task id and explicit goal position removed.
        This should be the default `model_obs` for GradMem/ICIL experiments.

    obs_positions_only_no_goal
        A stricter diagnostic variant. Keep end-effector/object state fields needed for control, but remove explicit goal/task fields.

Why remove goals and task IDs?
------------------------------
The purpose is to make the support demonstrations define the task. If the query observation contains a task ID or explicit goal vector, the query can solve the task without support. That defeats the point of GradMem-style memory writing.

Use Meta-World ML-style benchmarks when possible
-----------------------------------------------
Prefer ML-style Meta-World benchmarks over MT-style benchmarks for the memory-learning diagnostic.

Rationale:
- MT10/MT50 style settings can include explicit task IDs.
- ML1/ML10/ML45 style settings are closer to meta-learning and adaptation.

Implementation must still be configurable:

    benchmark = "ML1" | "ML10" | "ML45" | "MT10" | "MT50" | "custom"
    task_names = optional list[str]
    train_or_test = "train" | "test" where applicable
    remove_task_id = True by default
    remove_goal = True by default

If task IDs are present in a Meta-World observation, store them only as metadata and do not include them in `model_obs` unless the config explicitly requests it.

Task builder for query-memory MAML/FOMAML
-----------------------------------------
Implement a Meta-World analogue of `QueryMemoryTaskBuilder`.

Suggested classes:

    MetaWorldMAMLTaskSpec
    MetaWorldQueryMemoryTaskBuilder

The builder should produce the same conceptual structure as the RLBench query-memory task builder:

For JAX compatibility, the inner/query batch dictionaries should contain keys compatible with `icil_jax_query_memory/data/adapter.py`.

Because Meta-World is low-dimensional, there are two possible compatibility modes:

Mode A: state-only canonical mode
---------------------------------
Preferred long-term. Add a JAX adapter/model config option that consumes:

    query_state:    [B, T_obs, obs_dim]
    target_action:  [B, H, action_dim]

without requiring point-cloud keys.

This may require a small model-side change: add a state-only query encoder to `icil_jax_query_memory/models/`, e.g.

and allow `QueryMemoryDirectRegressionModel` to use it when `cfg.model.query_encoder_name = "state_only"`.

Mode B: compatibility shim
--------------------------
For immediate compatibility with the existing JAX adapter, create dummy point-cloud fields:

    query_xyz:   zeros [B, T_obs, 1, 3]
    query_valid: ones  [B, T_obs, 1]

and put the actual Meta-World observation into:

    query_state: [B, T_obs, obs_dim]

This allows existing code that expects `query_xyz`, `query_valid`, `query_state`, and `target_action` to run with minimal changes.

Implement Mode B first if it avoids modifying training code. Also add Mode A if easy, but do not block the data pipeline on it.

Target/action chunk construction
--------------------------------
Use the same high-level convention as RLBench ICIL:

    T_obs: number of query observation frames
    H: action chunk horizon
    stride: spacing between query observation frames and action targets

For a sampled starting time t0:

    query_obs_indices = [t0 - (T_obs-1)*stride, ..., t0]
    action_indices    = [t0 + 1, t0 + 2, ..., t0 + H]

or, if you want strict compatibility with the current RLBench stride convention, implement both:

    action_stride = 1 by default
    obs_stride = cfg.stride

Important: Do not repeat the earlier RLBench mismatch where action targets were stride-2 but eval executed them consecutively. For Meta-World direct-regression diagnostics, default to consecutive future actions:

    action_indices = t0 + 1 ... t0 + H

If an episode is too short, either reject the sample or pad the final action with the last valid action, controlled by config:

    cfg.dataset.pad_short_chunks: bool = False by default

Batch keys for JAX no-encoder GradMem path
------------------------------------------
For each inner/support batch:

    query_state:    [Q, T_obs, obs_dim]
    target_action:  [Q, H, action_dim]
    query_xyz:      [Q, T_obs, 1, 3] dummy zeros if compatibility mode
    query_valid:    [Q, T_obs, 1] dummy true if compatibility mode

For each query/read batch:

    query_state:    [Q_read, T_obs, obs_dim]
    target_action:  [Q_read, H, action_dim]
    query_xyz:      [Q_read, T_obs, 1, 3]
    query_valid:    [Q_read, T_obs, 1]

Prepared task batch iterable should support:

    K: number of support episodes
    num_queries_per_step: support chunks per WRITE step
    num_query_loss_samples: query chunks per READ/meta-loss
    inner_steps
    num_inner_batches

The first consumer is `icil_jax_query_memory/train/runner.py`, whose current adapter expands `inner_batches` according to `inner_steps` and stacks mandatory keys `query_xyz`, `query_state`, `query_valid`, and `target_action`.
