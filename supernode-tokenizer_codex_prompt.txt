================================================================================
0. PAPER STORY / CLAIMS
================================================================================

The paper story is:

> A locality-preserving supernode point-cloud tokenizer improves robot imitation
> learning, especially on geometry-sensitive manipulation tasks, under matched
> downstream policy heads and comparable compute.

The core hypothesis is:
- Perceiver-style global compression is efficient, but may lose fine local geometry.
- The supernode tokenizer preserves local structure before global compression.
- This improves policy performance on tasks requiring precise 3D grounding:
  grasp alignment, handles, inserts, local contact, and clutter.

The paper must isolate the effect of the tokenizer. Therefore:
- Keep the downstream policy head fixed whenever comparing tokenizers.
- Keep task conditioning fixed whenever comparing tokenizers.
- Keep training budget / data / evaluation protocol fixed whenever comparing tokenizers.

Do NOT turn this into a broad “new robot policy architecture” paper.
This is a tokenizer / representation paper.

5. TASK CONDITIONING (IMPORTANT)
================================================================================

This is simple class-conditioned IL, not language-conditioned IL in v1.
So the default conditioning is **task id embedding**, not a text encoder.

However, implement the conditioning stack cleanly and modularly so that a language embedding
backend could be plugged in later.

Use BOTH of the following simultaneously:

LEVEL 1 = early conditioning in the encoder/tokenizer
LEVEL 2 = late conditioning in the decoder/policy head

These are not alternatives. Use both.

5.1 Level 1 conditioning (encoder-side)
--------------------------------------
Goal: the task should influence what geometry is extracted from the point cloud.

Implement a `TaskConditioner` module that produces:
- `task_emb: [B, d_model]`
- `task_tokens: [B, n_task_tokens, d_model]`

For v1:
- task conditioning source = `nn.Embedding(num_tasks, d_model)`
- no text model yet

Encoder-side conditioning should use a standard strong recipe inspired by RT-1:
- identity-initialized FiLM / AdaLN-style modulation inside the encoder/tokenizer blocks
- where identity-init means the conditioning starts as a no-op
- do this in places where it is cheap and stable:
  - supernode update MLPs / point-to-supernode aggregation MLPs
  - optional Perceiver frame tokenizer internal blocks
  - post-tokenizer self-attention refiner blocks

At minimum:
- if full FiLM/AdaLN insertion is too invasive in v1, do additive conditioning to tokens as a fallback
- but the preferred implementation is identity-init FiLM/AdaLN

5.2 Level 2 conditioning (decoder-side)
--------------------------------------
Goal: the task should influence the action generation stage explicitly.

Use task tokens in the decoder memory:
- project `task_emb` into `n_task_tokens` learned tokens
- append them to the encoder memory tokens
- decoder cross-attends to `[obs_memory_tokens ; task_tokens]`

Defaults:
- `n_task_tokens = 4`
- task tokens are always present in all main experiments

Optionally:
- allow decoder AdaLN to consume `task_emb` as well, if this can be re-used cheaply from current DiT code
- keep this behind config and default ON if implementation is clean

7. COMMON POLICY HEADS / DECODERS
================================================================================

Two policy heads are needed:

PRIMARY = chunk regression transformer decoder
SECONDARY = diffusion transformer decoder

The main paper comparison should use the PRIMARY head.
The diffusion head is a secondary experiment to show the tokenizer gains are not specific to one head.

7.1 Primary policy head: Chunk regression decoder
------------------------------------------------
This is the main head for the paper.

Requirements:
- transformer-based action decoder
- same decoder shared across all tokenizers
- outputs an H-step action chunk in parallel
- loss = L1 by default
- optional diagonal Gaussian NLL behind config, but not needed for main paper

Design requirements (inspired by ACT / strong standard practice):
- use learned action query tokens of length H
- add learned or sinusoidal action-slot positional embeddings
- stack decoder blocks with:
  - self-attention over action query tokens
  - cross-attention to encoder memory tokens + task tokens
  - MLP / FFN
- no autoregressive decoding in the main head
- predict all H action slots in parallel
- output projection to action_dim

Strong preference:
- re-use the current DiT decoder block implementation as much as possible
- create a simplified non-diffusion variant instead of writing an entirely new decoder from scratch
- for example, adapt the current DiT block to a single-context standard transformer decoder block
- if there is existing AdaLN infrastructure that can consume task embeddings cleanly, reuse it

Default architecture for the chunk decoder:
- d_model = 512
- n_heads = 8
- n_layers = 8
- mlp_mult = 4
- dropout = 0.0
- action horizon H = 16
- task tokens appended to memory
- L1 loss as default objective

7.2 Secondary policy head: Diffusion decoder
--------------------------------------------
Implement a standard IL diffusion policy by adapting the current DiT diffusion code.

Requirements:
- single observation memory only (no two-context ICIL split)
- memory = observation tokens + task tokens
- reuse existing DiT decoder, scheduler, v-prediction setup as much as possible
- no support/demo branch
- no ICIL-specific losses

Default diffusion settings:
- reuse current repo defaults if they match the ICIL codepath:
  - DDIM or current scheduler path already implemented
  - v-prediction target if already present
  - H = 16
  - d_model = 512
  - n_heads = 8
  - 10 diffusion decoder layers (reuse current default if possible)
